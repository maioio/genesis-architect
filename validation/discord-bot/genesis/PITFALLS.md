## Pitfall 1: on_ready fires on every reconnect - sync() hits 60/day global rate limit

**Where:** https://github.com/Rapptz/discord.py/issues/7439
**Why:** Discord reconnects the gateway every few hours (network drop, Discord restart, rate limit). `on_ready` fires each time. `await bot.tree.sync()` on every reconnect consumes the global slash command sync quota (60/day for global commands, 2/10min for guild). Bot stops responding to slash commands silently after quota exhausted.
**Mitigation:** Guard on_ready with `hasattr(bot, "_ready_done")`. Run setup (sync, DB init) only once. Use `--sync` CLI flag for explicit sync in dev; never sync on prod restart.
**mitigation_file_path:** src/bot.py

**Implementation:**
- Create: src/bot.py with `on_ready` checking `if hasattr(bot, "_ready_done"): return` before any sync or setup, then `bot._ready_done = True`
- Test: tests/test_bot.py - call on_ready handler twice via mock, verify tree.sync called exactly once not twice
- Constrain: no `await bot.tree.sync()` inside on_ready without the hasattr guard
- Validate: `grep -n "tree.sync" src/bot.py` - must appear inside hasattr guard block

## Pitfall 2: No permission check - any member can run ban/kick

**Where:** https://github.com/Rapptz/discord.py/issues/8931 (pattern: missing permission decorator)
**Why:** Discord slash commands without `@app_commands.checks.has_permissions()` decorator execute for any guild member. A regular user who discovers `/ban` can ban other members (limited by their own Discord role, but the bot will attempt it and may succeed depending on role hierarchy). Also, the bot itself may lack the permission and crash silently.
**Mitigation:** Add `@app_commands.checks.has_permissions(ban_members=True)` to ban, `kick_members=True` to kick. Add error handler for `MissingPermissions` that responds with a user-facing error instead of propagating.
**mitigation_file_path:** src/commands/moderation.py

**Implementation:**
- Create: src/commands/moderation.py with all moderation commands, each decorated with `@app_commands.checks.has_permissions(...)` matching the required Discord permission
- Create: src/error_handler.py with `on_app_command_error` that catches `MissingPermissions` and responds with ephemeral "You don't have permission to use this command."
- Test: tests/test_permissions.py - mock interaction with a user lacking ban_members permission, verify response contains permission error, not "Banned"
- Constrain: no moderation command without `@app_commands.checks.has_permissions(...)` decorator

## Pitfall 3: purge without limit cap - interaction times out at 15s

**Where:** https://github.com/Rapptz/discord.py/issues/9428 (rate limit cascade pattern)
**Why:** `channel.purge(limit=amount)` with user-supplied amount. Discord bulk delete is capped at 100 messages per API call. Purging 10000 messages = 100 API calls = 200+ seconds. Discord interaction acknowledgment must happen within 3s (deferred to 15s max). Bot crashes with `InteractionResponded` or user sees "The application did not respond."
**Mitigation:** Cap purge at 100. Use `interaction.response.defer()` for operations that take >3s. Always respond within the 15s window or the interaction is dead.
**mitigation_file_path:** src/commands/moderation.py

**Implementation:**
- Create: `MAX_PURGE = 100` constant; reject amounts > MAX_PURGE with user-facing error before purge starts
- Test: tests/test_moderation.py - call clear with amount=10000, verify response is an error message (not a purge), verify channel.purge never called with limit > 100
- Constrain: no `channel.purge(limit=user_value)` without capping to MAX_PURGE first
- Validate: `grep -n "MAX_PURGE" src/commands/moderation.py` returns at least one match
