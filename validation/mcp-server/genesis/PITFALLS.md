## Pitfall 1: Tool inputSchema missing "required" field - args silently None

**Where:** https://github.com/modelcontextprotocol/python-sdk/issues/234
**Why:** MCP spec requires `required` array in inputSchema. Without it, Claude passes missing args as None instead of erroring. Tool receives `owner=None`, constructs URL `https://api.github.com/repos/None/None/issues`, gets 404, crashes with unhandled exception. No indication to the user why it failed.
**Mitigation:** Every tool schema must declare `required` listing all mandatory parameters. Add a schema validation test that rejects tools missing `required`.
**mitigation_file_path:** src/server.py

**Implementation:**
- Create: src/server.py with every tool inputSchema including `"required": ["param1", "param2"]`
- Test: tests/test_schema.py - import tool list, assert each tool's inputSchema contains "required" key with at least one entry
- Validate: `python -c "import json,sys; [sys.exit(1) for t in json.load(open('tools.json')) if 'required' not in t['inputSchema']]"`

## Pitfall 2: Large result set overflows Claude context window

**Where:** https://github.com/modelcontextprotocol/python-sdk/issues/198
**Why:** get_issues on active repos (fastapi, react) returns 30 issues * 2-5KB body each = 60-150KB in one tool result. Claude's context window fills, subsequent tool calls fail, or the MCP session hangs. No user-visible error - session just stops responding.
**Mitigation:** Cap all query results. Default max_rows=20, hard cap=50. Truncate issue body to 500 chars. Return a `truncated: true` field when capping. Let user pass `limit` parameter explicitly.
**mitigation_file_path:** src/limits.py

**Implementation:**
- Create: src/limits.py with `MAX_ISSUES = 20`, `BODY_PREVIEW_CHARS = 500`, `truncate_result(issues, limit)` that caps list and truncates body
- Test: tests/test_limits.py - pass 100 issues to truncate_result, verify len <= 20 and all body fields <= 500 chars; verify `truncated: true` in result
- Constrain: no direct return of full GitHub API response - always pass through truncate_result
- Validate: `grep -r "truncate_result" src/` returns at least one match per tool handler

## Pitfall 3: GitHub API 403/429 rate limit crashes server silently

**Where:** https://github.com/modelcontextprotocol/servers/issues/412 (pattern)
**Why:** urllib.request.urlopen raises HTTPError on 403/429. Unhandled exception propagates through MCP tool call handler, which crashes the entire server process (not just the one tool call). All subsequent tool calls fail until restart.
**Mitigation:** Wrap all external HTTP calls in try/except. On 403: return `isError: true` with "GitHub rate limit exceeded. Set GITHUB_TOKEN env var for 5000 req/hour." On 429: parse Retry-After header, return wait time to user.
**mitigation_file_path:** src/github_client.py

**Implementation:**
- Create: src/github_client.py with `fetch_json(url)` that catches HTTPError, returns `{"error": ..., "isError": True}` instead of raising; catches URLError for network failures
- Test: tests/test_github_client.py - mock urllib to raise HTTPError(403), verify fetch_json returns dict with isError=True and message containing "rate limit"; mock HTTPError(429) with Retry-After header, verify message contains retry time
- Constrain: no `urllib.request.urlopen` calls outside src/github_client.py
- Validate: `grep -rn "urlopen" src/` - only github_client.py should match
