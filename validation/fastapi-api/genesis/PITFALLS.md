# PITFALLS.md - FastAPI Task Management API
Generated by Genesis Architect | Research quality: PARTIAL

## Pitfall 1: Generator dependencies (DB sessions) close before middleware runs

**Where:** https://github.com/fastapi/fastapi/issues/11143
**Why:** After FastAPI 0.106, `yield`-based Depends (the standard DB session pattern) are torn down before HTTP middleware finishes. If middleware tries to commit/rollback after the route handler returns, the session is already closed. Silent data loss in apps that rely on middleware-level transaction management.
**Mitigation:** Use lifespan-scoped session management, not middleware-level transaction control. All commits/rollbacks must happen inside the route handler or the dependency itself.
**mitigation_file_path:** src/deps.py

**Implementation:**
- Create: src/deps.py with `get_db()` generator that commits on success and rolls back on exception, all inside the `yield` block - never relies on middleware to close
- Test: tests/test_deps.py - simulate route handler raising an exception mid-transaction, verify rollback called and session closed before response returns
- Constrain: no middleware that calls `session.commit()` or `session.rollback()` - all transaction control inside deps.py

## Pitfall 2: Default SQLAlchemy QueuePool exhausted under concurrent load

**Where:** https://github.com/PrefectHQ/prefect/issues/6492
**Why:** Default pool is size=5, max_overflow=10. Under 50+ concurrent requests, pool exhausts and requests hang indefinitely (no timeout configured). When pool is exhausted and connections time out, SQLAlchemy reuses a stale connection from the pool that was closed by Postgres, causing `asyncpg.InterfaceError: connection is closed`.
**Mitigation:** Configure pool explicitly: `pool_size=20, max_overflow=40, pool_timeout=30, pool_recycle=1800`. Always set `pool_pre_ping=True` to detect stale connections before use.
**mitigation_file_path:** src/database.py

**Implementation:**
- Create: src/database.py with `create_engine(DATABASE_URL, pool_size=20, max_overflow=40, pool_timeout=30, pool_recycle=1800, pool_pre_ping=True)` - all pool params explicit, never defaults
- Test: tests/test_database.py - mock 25 concurrent DB calls (above default pool size of 5), verify no InterfaceError and all requests complete
- Constrain: no `create_engine(url)` with only the URL - pool params must always be specified explicitly

## Pitfall 3: Alembic auto-upgrade not applied on startup - table missing crash

**Where:** https://github.com/open-webui/open-webui/issues/21349
**Why:** FastAPI lifespan handler does not run `alembic upgrade head` automatically. When a new table is added in a migration and the app is deployed without running migrations, SQLAlchemy raises `OperationalError: no such table` on the first query, crashing the app with a 500 that looks unrelated to migrations.
**Mitigation:** Run `alembic upgrade head` in the lifespan startup handler before accepting requests. Log the current revision before and after. If upgrade fails, refuse to start.
**mitigation_file_path:** src/main.py

**Implementation:**
- Create: lifespan handler in src/main.py that calls `subprocess.run(["alembic", "upgrade", "head"], check=True)` before `yield`, wrapped in try/except that logs and re-raises on failure (startup abort)
- Test: tests/test_startup.py - mock alembic upgrade to fail, verify app refuses to start (lifespan raises, not swallowed)
- Constrain: no SQLAlchemy `Base.metadata.create_all()` as a migration strategy in production code - Alembic only

## Pitfall 4: Async pytest-asyncio scope mismatch silently skips tests

**Where:** https://github.com/pytest-dev/pytest-asyncio/issues/706
**Why:** pytest-asyncio 0.23 changed default asyncio_mode to strict. Session-scoped async fixtures that worked in 0.21 silently pass (0 assertions run) in 0.23 when `asyncio_mode` is not declared in pyproject.toml. Tests appear green but nothing is tested.
**Mitigation:** Pin `pytest-asyncio>=0.23` and add `asyncio_mode = "auto"` to `[tool.pytest.ini_options]` in pyproject.toml. Add a canary test that asserts a real DB call returns data - if it passes with 0 rows, it fails.
**mitigation_file_path:** pyproject.toml

**Implementation:**
- Create: pyproject.toml with `[tool.pytest.ini_options]` section containing `asyncio_mode = "auto"` and `pytest-asyncio>=0.23` in dev dependencies
- Test: tests/test_canary.py - a test that creates one task and asserts `len(tasks) == 1` (not just `>= 0`), ensuring async fixtures actually ran
- Constrain: no `@pytest.mark.asyncio` on individual tests when asyncio_mode=auto is set (redundant and masks scope issues)
